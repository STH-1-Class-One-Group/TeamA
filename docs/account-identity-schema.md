# Account / Identity / Delete Rules

## 1. 내부 계정과 외부 로그인 분리

JamIssue는 내부 서비스 계정과 소셜 로그인 식별자를 분리합니다.

### user
- 서비스 내부 고유 계정
- 모든 피드, 댓글, 스탬프, 경로는 `user.user_id` 를 참조

### user_identity
- 외부 로그인 연결 정보
- `provider`
- `provider_user_id`
- `user_id`

즉, 네이버와 카카오는 하나의 `user_id` 아래에 여러 identity로 연결될 수 있습니다.

## 2. 같은 이메일 자동 병합 금지

같은 이메일이라고 해서 네이버/카카오 계정을 자동으로 합치지 않습니다.

이유:
- 제공자마다 이메일 정책이 다를 수 있음
- 공유 메일 / 임시 메일 / 이메일 누락 가능성 존재
- 잘못된 자동 병합은 계정 탈취와 비슷한 사고로 이어질 수 있음

따라서:
- 기본 로그인은 제공자별 별도 identity
- 계정 통합은 명시적 연결일 때만 수행

## 3. 회원 탈퇴 규칙

회원 탈퇴는 `user_id` 기준으로 정리합니다.

정리 대상:
- `user_identity`
- `feed`
- `feed_like`
- `user_comment`
- `user_stamp`
- `travel_session`
- `user_route`
- `user_route_like`

효과:
- 내부 계정 기준으로 기록이 정리됨
- 외부 로그인 수단이 바뀌어도 삭제 기준은 흔들리지 않음

## 4. 댓글 삭제 규칙

### 댓글 직접 삭제
- hard delete가 아니라 `is_deleted = true`
- 본문은 비우고, 대댓글 문맥은 유지
- 화면에는 `삭제된 댓글입니다.` 로 표시

### 피드 삭제
- 피드가 삭제되면 해당 피드 아래 댓글/좋아요는 함께 정리

### parent_id 규칙
- `user_comment.parent_id` 는 self FK
- `ON DELETE SET NULL`
- 부모 댓글이 사라져도 자식 댓글은 유지 가능

## 5. 왜 이 구조가 PRD에 맞는가

- 로그인 수단보다 서비스 내부 기록을 우선 유지
- 사용자의 방문 기록, 후기, 추천 경로가 인증 제공자에 종속되지 않음
- 삭제와 관계형 무결성이 분리되지 않고 함께 맞물림
- 이후 네이버/카카오 확장에도 데이터가 덜 흔들림
